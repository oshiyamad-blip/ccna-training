## 1. DHCP — アドレスの自動割り当てと DORA

### DHCP とは何か

**DHCP（Dynamic Host Configuration Protocol）** は、ネットワークに接続したホストへ
**IP アドレス・サブネットマスク・デフォルトゲートウェイ・DNS サーバ・リース時間**
などの設定情報を自動的に配布するプロトコルです。手動での IP アドレス設定作業を
不要にし、設定ミスやアドレス重複を防ぐ効果があります。

DHCP は UDP を使用し、**サーバ側は UDP 67**、**クライアント側は UDP 68** の
ポート番号で通信します。

> **試験のポイント**: DHCP サーバ = UDP 67、クライアント = UDP 68 というポート番号の
> 組み合わせを問う問題が頻出です。逆に覚えないよう注意してください。

### DORA の 4 段階

DHCP によるアドレス取得は、次の 4 つのメッセージのやり取りで行われます。頭文字を
取って **DORA** と呼ばれます。

| 順序 | メッセージ | 送信元 → 宛先 | 内容 |
|---|---|---|---|
| ① | **D**iscover | クライアント → ブロードキャスト（送信元 `0.0.0.0`、宛先 `255.255.255.255`） | 「DHCP サーバはいますか」とネットワーク全体に問いかける |
| ② | **O**ffer | サーバ → クライアント | 提供可能な IP アドレス等の設定情報を提示する |
| ③ | **R**equest | クライアント → ブロードキャスト | 提示されたアドレスの使用を要求する（複数サーバから Offer を受けた場合、選んだ 1 台への要求を全体へブロードキャストすることで他サーバへ辞退を伝える） |
| ④ | **A**ck | サーバ → クライアント | 要求を承認し、リースを確定する |

```
クライアント                          サーバ
     │── Discover（ブロードキャスト）──▶│
     │◀──────── Offer ─────────────────│
     │── Request（ブロードキャスト）───▶│
     │◀──────── Ack ────────────────────│
```

クライアントはまだ自分の IP アドレスを持っていないため、Discover の送信元は
`0.0.0.0`、宛先は限定ブロードキャストの `255.255.255.255` になります。

> **試験のポイント**: DORA の順序（Discover → Offer → Request → Ack）と、
> 各メッセージの送信元・宛先アドレスを問う問題が頻出です。

### リースの更新（T1・T2）

DHCP によって割り当てられたアドレスには**リース時間**（有効期限）があります。
期限切れ間近になると、クライアントは次の 2 段階でリースの更新を試みます。

- **T1（リースの 50% 経過時点）**: 元々アドレスを割り当てたサーバへ**ユニキャスト**で
  更新（Request）を要求する
- **T2（リースの 87.5% 経過時点）**: T1 での応答がなかった場合、任意の DHCP サーバへ
  **ブロードキャスト**で再バインドを試みる

T1・T2 とも応答が得られなければ、リース満了時点でアドレスは失効します。

### ルータを DHCP サーバとして設定する

Cisco ルータは自身を DHCP サーバとして動作させることができます。設定は次の手順です。

1. **固定用に使うアドレスを除外**する（サーバ・ルータ自身・プリンタなど、
   DHCP で配布したくないアドレス範囲を先に除外しておく）

   ```
   Router(config)# ip dhcp excluded-address 192.168.1.1 192.168.1.10
   ```

2. **DHCP プールを作成**し、配布するパラメータを設定する

   ```
   Router(config)# ip dhcp pool LAN1
   Router(dhcp-config)# network 192.168.1.0 255.255.255.0
   Router(dhcp-config)# default-router 192.168.1.1
   Router(dhcp-config)# dns-server 192.168.1.10
   Router(dhcp-config)# lease 0 8 0
   ```

   - `network` — 配布するサブネットの範囲
   - `default-router` — クライアントに配布するデフォルトゲートウェイ
   - `dns-server` — クライアントに配布する DNS サーバアドレス
   - `lease` — リース時間（日 時 分の順。例の `0 8 0` は 8 時間）。省略時の既定値は 1 日

> **試験のポイント**: `ip dhcp excluded-address` を**プールの外（グローバル
> コンフィグモード）**で先に設定しておかないと、除外したいアドレスまで
> クライアントに配布されてしまう点に注意してください。

### 検証コマンド

| コマンド | 用途 |
|---|---|
| `show ip dhcp binding` | 配布済みのリース（クライアントの MAC・IP・リース期限）を確認 |
| `show ip dhcp pool` | プールごとの使用状況（配布数・残り件数）を確認 |
| `show ip dhcp conflict` | アドレス重複（コンフリクト）の検出履歴を確認 |

### DHCP リレー（ip helper-address）

DHCP のリクエスト（Discover・Request）は**ブロードキャスト**で送られますが、
**ルータは既定でブロードキャストを他のセグメントへ転送しません**。そのため、
クライアントと DHCP サーバが**別のサブネット**にある場合、そのままではクライアントの
Discover がサーバまで届かず、アドレスを取得できません。

この問題を解決するのが **DHCP リレーエージェント**です。クライアント側の
インターフェースに次のコマンドを設定します。

```
Router(config)# interface GigabitEthernet0/0
Router(config-if)# ip helper-address 192.168.1.10
```

`ip helper-address <サーバ IP>` を設定すると、そのインターフェースで受信した
DHCP のブロードキャストを、指定した DHCP サーバ宛の**ユニキャスト**に変換して
転送してくれます。サーバからの応答も、リレーしたルータを経由してクライアントへ
戻されます。

> **試験のポイント**: なぜサブネットをまたぐ DHCP に `ip helper-address` が
> 必要なのか（DHCP Discover がブロードキャストであり、ルータはブロードキャストを
> 転送しないため）を説明させる問題が非常によく出題されます。

### ip helper-address が転送する既定のサービス

`ip helper-address` は DHCP（UDP 67/68）専用ではなく、既定で次のような
複数の UDP サービスのブロードキャストをまとめて転送します。

| サービス | ポート |
|---|---|
| DNS | UDP 53 |
| **DHCP（Bootp）** | **UDP 67 / 68** |
| TFTP | UDP 69 |
| TIME | UDP 37 |
| TACACS | UDP 49 |
| NetBIOS（Name Service / Datagram Service） | UDP 137 / 138 |

### DHCP クライアントとしてのルータ設定

ルータ自身のインターフェースを DHCP クライアントとして動作させ、上位（ISP など）
から自動的に IP アドレスを取得することもできます。

```
Router(config)# interface GigabitEthernet0/1
Router(config-if)# ip address dhcp
```

## 2. DNS — 名前解決の役割

### DNS とは何か

**DNS（Domain Name System）** は、`www.example.com` のような**人間が覚えやすい
ホスト名**を、通信に必要な **IP アドレス**へ変換（**名前解決**）する分散データベース
サービスです。DNS が使用するポート番号は、通常の問い合わせでは **UDP 53**、
ゾーン転送のような大きなデータのやり取りでは **TCP 53** が使われます。

### 主な DNS レコード

| レコード種別 | 用途 |
|---|---|
| **A** | ホスト名 → IPv4 アドレス |
| **AAAA** | ホスト名 → IPv6 アドレス |
| **CNAME** | 別名（エイリアス）から正式なホスト名へのマッピング |
| **MX** | ドメイン宛のメールを処理するメールサーバの指定（優先度付き） |
| **NS** | そのドメインを管理するネームサーバの指定 |
| **PTR** | 逆引き。IP アドレス → ホスト名 |

> **試験のポイント**: A（IPv4）・AAAA（IPv6）・CNAME（別名）・MX（メール）・
> NS（ネームサーバ）・PTR（逆引き）の役割を対応させて覚えておく必要があります。

### DNS の階層構造

DNS はツリー状の**階層構造**を持つ分散データベースです。

```
ルート（.）
  └─ TLD（トップレベルドメイン：.com、.jp など）
        └─ 権威 DNS サーバ（example.com などの各ドメインを管理）
```

クライアントからの問い合わせには 2 つの方式があります。

- **再帰問い合わせ（Recursive Query）**: 問い合わせを受けたサーバが、
  必要な情報を持つサーバを自ら順にたどり、**最終的な答えだけをクライアントへ返す**
- **反復問い合わせ（Iterative Query）**: 問い合わせを受けたサーバは、
  自分が知らなければ「次はこのサーバに聞いてください」という**参照情報を返す**
  （クライアント側や上位のリゾルバが次のサーバへ改めて問い合わせる）

一般に、クライアント（PC）からローカル DNS サーバへの問い合わせは再帰問い合わせ、
ローカル DNS サーバから上位（ルート〜権威）への問い合わせは反復問い合わせで
行われることが多いです。

### ルータを DNS クライアントとして設定する

```
Router(config)# ip domain-name example.local
Router(config)# ip name-server 192.168.1.10
Router(config)# ip domain-lookup
```

- `ip name-server <DNS サーバ IP>` — 問い合わせ先の DNS サーバを指定
- `ip domain-name <ドメイン名>` — ルータに割り当てるドメイン名を指定（SSH の鍵生成にも必要）
- `ip domain-lookup` — DNS による名前解決機能そのものの有効化（**既定で有効**）

### 静的な名前解決（ip host）

DNS サーバを使わずに、ルータのローカル設定だけで名前と IP アドレスの対応表を
持たせることもできます。

```
Router(config)# ip host R2 10.0.0.2
```

これ以降、そのルータ上で `ping R2` のように**名前で直接指定**できるようになります。

### ip domain-lookup が有効なままだと起こる問題

`ip domain-lookup` が有効な状態で、特権 EXEC モードなどでコマンドを**打ち間違える**と、
IOS はそれをホスト名だと解釈し、DNS サーバへ名前解決を試みてから
「Unknown command」と表示するまで**しばらく待たされる**ことがあります
（DNS サーバが設定されていない、または応答がない場合に顕著です）。

この挙動を避けるため、実運用やラボ環境では次のコマンドで名前解決機能自体を
無効化しておく運用がよく行われます。

```
Router(config)# no ip domain-lookup
```

> **試験のポイント**: `no ip domain-lookup` を設定する理由（コマンドの打ち間違いを
> ホスト名と誤認し、名前解決の待ち時間が発生するのを防ぐため）を問う問題が
> 出題されることがあります。

## 3. NTP — 時刻同期と階層（stratum）

### NTP とは何か

**NTP（Network Time Protocol）** は、ネットワーク機器同士の時刻を高精度に
同期させるためのプロトコルで、**UDP 123** を使用します。ログの相関分析
（複数機器のログを時刻で突き合わせる）、デジタル証明書の有効期限検証、
Kerberos 認証など、正確な時刻はネットワーク運用の多くの場面で不可欠です。

### stratum（階層）の考え方

**stratum** は、その時刻情報が「基準となる時刻源からどれだけ離れているか」を
表す階層番号です。

| stratum | 意味 |
|---|---|
| **stratum 0** | 原子時計・GPS 受信機など、基準となる参照クロックそのもの（ネットワーク機器ではない） |
| **stratum 1** | stratum 0 に直結され、時刻を配信するサーバ |
| stratum 2, 3, … | 上位の stratum からさらに時刻を受け取って配信するサーバ（数値が大きいほど基準から遠い） |
| **stratum 16** | **未同期**（有効な時刻源を持たない）状態を表す特別な値 |

stratum の最大値は **15** で、**16 は「同期していない」ことを意味する特別な値**
です。

> **試験のポイント**: stratum 0 が参照時刻源そのもの（機器ではない）であること、
> 数値が大きいほど基準から遠いこと、16 は未同期を意味することが繰り返し
> 問われます。

### ルータを NTP マスタ・クライアントにする

ルータ自身を基準時刻源（stratum の起点）として動作させるには、次のコマンドを
使用します。

```
Router(config)# ntp master 3
```

これにより、このルータは自身を stratum 3 として振る舞い、他の機器へ時刻を配信します。

他のルータからこのマスタを参照（同期）させるには、クライアント側で次のように
設定します。

```
Router(config)# ntp server 10.0.0.1
```

### 検証コマンド

| コマンド | 用途 |
|---|---|
| `show ntp status` | 自身の同期状態（synchronized か否か）、stratum、参照先のアドレスなどを表示 |
| `show ntp associations` | ピア（参照先候補）の一覧を表示。先頭に `*` が付いているピアが、実際に**同期中**の参照先 |

### NTP 認証

不正な機器から偽の時刻情報を受け取らないよう、**NTP 認証**を設定して信頼できる
サーバとのみ同期させることができます。

```
Router(config)# ntp authenticate
Router(config)# ntp authentication-key 1 md5 CCNA-KEY
Router(config)# ntp trusted-key 1
```

- `ntp authenticate` — NTP 認証機能自体を有効化
- `ntp authentication-key` — 認証に使う鍵番号・方式・パスワード（キー文字列）を定義
- `ntp trusted-key` — その鍵番号を信頼済み（実際に認証に使う）として指定

### 手動での時刻・タイムゾーン設定

NTP を使わない場合や初期設定として、手動で時刻を設定することもできます。

```
Router# clock set 10:00:00 13 July 2026
Router(config)# clock timezone JST 9
```

`clock set` は特権 EXEC モードで実行する時刻の即時設定、`clock timezone` は
グローバルコンフィグモードで設定するタイムゾーンのオフセットです。

## 4. SNMP と Syslog — 監視とログ収集

### SNMP とは何か

**SNMP（Simple Network Management Protocol）** は、ネットワーク管理システム
（NMS、マネージャ）がネットワーク機器（エージェント）を**監視・管理**するための
プロトコルです。

| 通信の方向 | ポート | 内容 |
|---|---|---|
| マネージャ → エージェント（ポーリング） | **UDP 161** | 情報の取得・設定要求 |
| エージェント → マネージャ（**Trap**） | **UDP 162** | エージェント側で発生したイベントの自発的な通知 |

> **試験のポイント**: SNMP のポーリングが UDP 161、Trap（通知）が UDP 162 という
> ポート番号の対応が頻出です。

### SNMP のバージョン

| バージョン | 認証方式 | セキュリティ |
|---|---|---|
| **SNMPv1 / v2c** | **コミュニティストリング**（`read-only` または `read-write`） | **平文**で送信され、盗聴・改ざんに弱い |
| **SNMPv3** | ユーザベースの認証 | **認証と暗号化**に対応する |

SNMPv3 では、次の 3 段階の**セキュリティレベル**を選択できます。

| セキュリティレベル | 認証 | 暗号化 |
|---|---|---|
| **noAuthNoPriv** | なし | なし |
| **authNoPriv** | あり | なし |
| **authPriv** | あり | あり |

> **試験のポイント**: v1/v2c はコミュニティストリングを平文送信するのに対し、
> v3 は認証・暗号化に対応するというセキュリティ面の違いが頻出です。

### MIB と OID、基本操作

- **MIB（Management Information Base）**: SNMP エージェントが持つ管理対象データを
  階層構造で整理したデータベース
- **OID（Object Identifier）**: MIB 内の各データ項目を一意に指し示す識別子
  （ドット区切りの数値列で表される）

SNMP の基本操作は次の 5 つです。

| 操作 | 主体 | 内容 |
|---|---|---|
| **Get** | マネージャ | 特定の OID の値を 1 つ取得する |
| **GetNext** | マネージャ | 次の OID の値を取得する（MIB を順に走査する際に使用） |
| **Set** | マネージャ | エージェント側の値を変更する |
| **Trap** | エージェント | イベント発生をマネージャへ自発的に通知する（応答なし） |
| **Inform** | エージェント | Trap と同様の通知だが、マネージャからの**確認応答（Ack）を要求**する |

### Syslog とは何か

**Syslog** は、ネットワーク機器が発生させたイベントログを収集・記録するための
仕組みです。ログには**重大度（severity）**が付与され、**0〜7 の数値**で表されます。
severity の値は**数値が小さいほど重大**です。

| severity | 名称 |
|---|---|
| **0** | Emergency（緊急） |
| **1** | Alert（警報） |
| **2** | Critical（重大） |
| **3** | Error（エラー） |
| **4** | Warning（警告） |
| **5** | Notification（通知） |
| **6** | Informational（情報） |
| **7** | Debugging（デバッグ） |

> **試験のポイント**: severity の 0〜7 の名称と、「数値が小さいほど重大」という
> 対応関係は非常によく出題されます。「大きいほど重大」と誤って覚えないよう
> 注意してください。

### ログの出力先

Syslog メッセージは複数の宛先に同時に出力できます。

| コマンド | 出力先 |
|---|---|
| `logging console` | コンソールポート |
| `logging monitor` | VTY（Telnet/SSH）セッション |
| `logging buffered` | ルータのメモリ上のログバッファ |
| `logging host <サーバ IP>` | 外部の Syslog サーバ |

Syslog サーバへ送信するメッセージの**最低 severity**（そのレベル以上を送る）は
次のコマンドで指定します。

```
Router(config)# logging host 192.168.1.10
Router(config)# logging trap informational
```

`logging trap informational` は「severity 6（Informational）以上の重大度
（つまり 0〜6）のメッセージをすべて送信する」ことを意味します。

### メッセージの形式とタイムスタンプ

Syslog メッセージの基本形式は次のとおりです。

```
seq: *timestamp: %FACILITY-SEVERITY-MNEMONIC: 説明
```

例えば、インターフェースがリンクアップしたときの代表的なメッセージは
次のようになります。

```
*Jul 13 10:15:32.123: %LINK-3-UPDOWN: Interface GigabitEthernet0/1, changed state to up
```

- `%LINK` — FACILITY（発生元の機能分類）
- `3` — SEVERITY（この例では 3 = Error）
- `UPDOWN` — MNEMONIC（メッセージの種類を示す短い識別子）

メッセージにタイムスタンプ（日時）を付与するには、次のコマンドを設定します。

```
Router(config)# service timestamps log datetime msec
```

### 検証コマンド

`show logging` を実行すると、現在のロギング設定（出力先・severity のしきい値等）と、
バッファに蓄積されたログの履歴を確認できます。

## 5. セキュアな管理アクセスとファイル転送（SSH / Telnet / FTP / TFTP）

### Telnet と SSH の違い

| 項目 | Telnet | SSH（Secure Shell） |
|---|---|---|
| ポート番号 | **TCP 23** | **TCP 22** |
| 通信内容 | **平文**（パスワードも含めすべて盗聴可能） | **暗号化**される |
| 推奨度 | 非推奨 | 管理アクセスには **SSHv2** の使用を推奨 |

> **試験のポイント**: Telnet（TCP 23・平文）と SSH（TCP 22・暗号化）のポート番号と
> セキュリティ特性の対比は頻出です。

### SSH の設定手順

SSH を有効化するには、次の順序で設定を行います。

1. **ホスト名を設定する**（RSA 鍵の生成に必要）

   ```
   Router(config)# hostname R1
   ```

2. **ドメイン名を設定する**（RSA 鍵の生成に必要）

   ```
   R1(config)# ip domain-name example.local
   ```

3. **RSA 鍵ペアを生成する**（この時点で暗号処理の基盤ができ、SSH が有効になる）

   ```
   R1(config)# crypto key generate rsa
   ```

   鍵のビット数を尋ねられたら **1024 ビット以上**を指定します（SSHv2 で動作させるには
   **768 ビット以上**が必要です）。

4. **ローカルユーザアカウントを作成する**

   ```
   R1(config)# username admin secret Cisco123!
   ```

5. **VTY 回線で SSH のみを許可し、ローカル認証を使う**

   ```
   R1(config)# line vty 0 4
   R1(config-line)# transport input ssh
   R1(config-line)# login local
   ```

6. **SSH のバージョンを 2 に固定する**

   ```
   R1(config)# ip ssh version 2
   ```

`transport input ssh` を設定すると、VTY 回線への Telnet 接続が拒否され、SSH のみが
許可されます。

> **試験のポイント**: SSH 有効化に必要な設定の流れ（`hostname` → `ip domain-name` →
> `crypto key generate rsa` → `username` → `line vty` での `login local` /
> `transport input ssh`）を問う問題が頻出です。手順の一部が欠けると SSH は
> 動作しません。

### TFTP と FTP

| プロトコル | ポート | 認証 | 特徴 |
|---|---|---|---|
| **TFTP**（Trivial File Transfer Protocol） | **UDP 69** | なし | 軽量・シンプル。IOS イメージや設定ファイルの転送によく使われる |
| **FTP**（File Transfer Protocol） | **TCP 20 / 21** | あり | 制御用（21）とデータ転送用（20）にポートを分けて使用する |

### 設定ファイルのバックアップ・復元

```
Router# copy running-config tftp
Router# copy tftp running-config
Router# copy startup-config tftp
```

- `copy running-config tftp` — 現在の実行中設定を TFTP サーバへバックアップ
- `copy tftp running-config` — TFTP サーバ上の設定ファイルを実行中設定へ読み込む（**マージ**され、既存の設定を上書きするとは限らない点に注意）
- `copy startup-config tftp` — 起動時設定を TFTP サーバへバックアップ

### IOS イメージのバックアップ・アップグレード

```
Router# copy flash tftp
Router# copy tftp flash
```

- `copy flash tftp` — ルータの flash メモリ上の IOS イメージを TFTP サーバへバックアップ
- `copy tftp flash` — TFTP サーバから新しい IOS イメージを flash へ書き込む（アップグレード）

### SCP による安全なファイル転送

**SCP（Secure Copy Protocol）** は SSH の暗号化通信の上で動作する、安全な
ファイル転送方式です。ルータを SCP サーバとして動作させるには、次のコマンドで
有効化します（SSH が事前に設定されていることが前提です）。

```
Router(config)# ip scp server enable
```

## 6. QoS の概要 — 分類・マーキング・キューイング・ポリシング/シェーピング

### QoS が必要な理由

**QoS（Quality of Service）** は、帯域が輻輳（混雑）した際に、音声・映像のような
**遅延やパケット損失に敏感なトラフィックを優先的に転送する**ための一連の仕組みです。
特に音声（VoIP）トラフィックの品質目安は次のとおりです。

| 指標 | 目安 |
|---|---|
| 片道遅延（Latency） | 150 ミリ秒未満 |
| ジッタ（遅延のばらつき） | 30 ミリ秒未満 |
| パケット損失 | 1% 未満 |

### 分類とマーキング

- **分類（Classification）**: トラフィックの種類を識別する処理。ACL（アクセスリスト）や
  **NBAR**（Network Based Application Recognition：アプリケーション単位での識別）、
  受信インターフェースなどを基準に判定する
- **マーキング（Marking）**: 分類した結果を、パケットやフレームの**ヘッダに書き込む**処理

マーキングには、階層に応じて次の 2 種類があります。

| 階層 | マーキング項目 | ビット数 | 備考 |
|---|---|---|---|
| **L2** | **CoS**（Class of Service） | 802.1Q タグ内の **3 ビット** | VLAN タグ付きフレームでのみ有効 |
| **L3** | **DSCP**（Differentiated Services Code Point）／ IP precedence | IP ヘッダの ToS フィールド内 **6 ビット**（DSCP） | ルーティングをまたいでも維持されやすい |

DSCP の代表的な値に、音声トラフィック向けの **EF（Expedited Forwarding、
DSCP 46）** があります。このほか **AF（Assured Forwarding）クラス**や
**CS（Class Selector）クラス**も定義されています。

> **試験のポイント**: L2 の CoS（3 ビット）と L3 の DSCP（6 ビット）の区別、
> および音声に使われる EF = DSCP 46 という対応が頻出です。

### キューイング（輻輳管理）

**キューイング**は、送信待ちのパケットを複数のキュー（待ち行列）に振り分け、
優先順位に応じて送出順序を制御する仕組みです。音声トラフィックのような
遅延に敏感なトラフィックには、他のキューより優先的に処理される
**LLQ（Low Latency Queuing）**、いわゆる**プライオリティキュー**を割り当てるのが
一般的です。

### ポリシングとシェーピング

契約帯域（レート）を超過したトラフィックへの対処方法として、
**ポリシング**と**シェーピング**の 2 つがあります。

| 項目 | ポリシング（Policing） | シェーピング（Shaping） |
|---|---|---|
| 超過分の扱い | **破棄**または**再マーク**する | **バッファに貯めて遅延**させ、送出する |
| バッファリング | 行わない（即時判定） | 行う |
| 主な適用方向 | 入力・出力の**どちらにも**適用可能 | 通常は**出力方向**に適用 |
| トラフィックの平滑化 | しない（バーストは急激に切り捨てられる） | する（レートが滑らかになる） |

> **試験のポイント**: ポリシングは超過分を破棄・再マークしバッファしない、
> シェーピングは超過分をバッファして遅延させる、という違いを問う問題が
> 頻出です。「どちらも破棄する」「どちらもバッファする」といった誤った
> 選択肢に注意してください。

### トラストバウンダリ（信頼境界）

**トラストバウンダリ（Trust Boundary）** とは、ネットワーク内で「どこまでの
機器（端末側）が行ったマーキングを信頼するか」を決める境界線のことです。
一般的には、**IP 電話機までの範囲を信頼する**（IP 電話機によるマーキングは
信頼するが、その先の一般 PC によるマーキングは信頼しない、あるいは
再マーキングする）構成が典型的です。信頼できない端末が不正に高い優先度を
マーキングしてしまうことを防ぐ目的があります。

## まとめ

- DHCP は DORA（Discover → Offer → Request → Ack）でアドレスを配布する。サーバ = UDP 67、クライアント = UDP 68
- 別サブネットへの DHCP 配布には `ip helper-address` によるリレーが必要（DHCP はブロードキャストで、ルータは転送しないため）
- DNS は UDP/TCP 53 を使い、A・AAAA・CNAME・MX・NS・PTR などのレコードで名前解決を行う
- NTP は UDP 123 を使用し、stratum（0 が基準源、16 が未同期）で階層を表す。`ntp master` / `ntp server` を使い分ける
- SNMP はポーリングが UDP 161、Trap が UDP 162。v1/v2c は平文コミュニティ、v3 は認証・暗号化に対応
- Syslog の severity は 0（Emergency）〜7（Debugging）で、数値が小さいほど重大
- SSH（TCP 22・暗号化）は Telnet（TCP 23・平文）に代わる安全な管理アクセス手段。有効化には `hostname` → `ip domain-name` → `crypto key generate rsa` → `username` → `line vty` の順の設定が必要
- QoS はマーキング（L2=CoS、L3=DSCP）で識別したトラフィックを、キューイングで優先制御し、ポリシング（破棄・再マーク・バッファなし）やシェーピング（バッファして遅延）で帯域を制御する

## 確認問題（自己チェック・解答は末尾）

1. DHCP のサーバ側とクライアント側が使用する UDP ポート番号をそれぞれ答えよ。
2. 別サブネットにある DHCP サーバへクライアントの Discover を届けるために、ルータのクライアント側インターフェースへ設定するコマンドは何か。
3. NTP における stratum 0 と stratum 16 は、それぞれ何を意味するか。
4. Syslog の severity レベルのうち、数値が最も小さいものと最も大きいものの名称をそれぞれ答えよ。
5. QoS のポリシングとシェーピングの、超過トラフィックに対する処理方法の違いを 1〜2 文で説明せよ。

<details><summary>解答</summary>

1. サーバ = UDP 67、クライアント = UDP 68
2. `ip helper-address <DHCP サーバの IP アドレス>`
3. stratum 0 は原子時計・GPS 等の基準となる参照クロックそのもの。stratum 16 は同期していない（未同期）状態を表す特別な値
4. 最も小さい（最も重大）のは 0 = Emergency、最も大きい（最も軽微）のは 7 = Debugging
5. ポリシングは超過分を即座に破棄または再マークし、バッファには貯めない。シェーピングは超過分をバッファに貯めて送出を遅延させることで、送出レートを平滑化する。

</details>
