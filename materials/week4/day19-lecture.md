## 1. コントローラベースネットワーキングと SDN の基礎

これまでの Day で学んできたルータやスイッチの設定は、基本的に 1 台ずつ CLI
（コマンドラインインターフェース）にログインして行うものでした。この方式を
**従来型（分散型）ネットワーク**と呼びます。各機器が自分自身で経路計算や転送判断を
自律的に行うため、構成はシンプルですが、機器の台数が増えるほど「1 台ずつ設定する」
作業量とヒューマンエラー（設定ミス）のリスクが増大します。

この課題を解決する考え方が **SDN（Software-Defined Networking、ソフトウェア定義
ネットワーキング）** です。SDN では、ネットワークの機能を次の 2 つの平面（プレーン）
に分けて考えます。

| 平面 | 役割 | 従来型 | SDN |
|---|---|---|---|
| **コントロールプレーン** | 経路をどう転送するか「決める」頭脳の部分 | 各機器が個別に保持 | コントローラに集中 |
| **データプレーン** | 決まった経路に従って実際にパケットを「転送する」部分 | 各機器 | 各機器（変わらず） |

SDN では、コントロールプレーンの機能を**コントローラ**と呼ばれる集中管理システムに
集約し、データプレーン（実際の転送）だけを各機器に残します。これにより、管理者は
個々の機器を意識せず、コントローラに対して指示を出すだけでネットワーク全体を
制御できるようになります。

### 南向き・北向きインターフェース

コントローラは 2 方向のインターフェースを持ちます。

- **南向きインターフェース（Southbound Interface: SBI）**: コントローラが配下の
  ネットワーク機器を制御するためのインターフェース。設定の配布や状態の取得に使う。
  例: NETCONF、RESTCONF、OpenFlow
- **北向きインターフェース（Northbound Interface: NBI）**: コントローラの上位に
  あるアプリケーションやオーケストレーションツール、自動化スクリプトがコントローラを
  操作するための入口。多くの場合 REST API の形をとる

```
[ 自動化スクリプト / オーケストレーションツール ]
          │  NBI（例: REST API）
          ▼
     [ コントローラ ]  ← コントロールプレーン集中
          │  SBI（例: NETCONF/RESTCONF, OpenFlow）
          ▼
[ ルータ / スイッチ ... ]  ← データプレーン（転送のみ）
```

### 集中管理の利点

- **設定の一貫性**: 全機器に同じポリシーを一括適用できる
- **迅速なプロビジョニング**: 新規機器の追加や設定変更を短時間で行える
- **ポリシーの一元適用**: セキュリティやQoSのルールを 1 か所で管理できる
- **可視化**: ネットワーク全体の状態を 1 つの画面で把握できる

> **試験のポイント**: CCNA では SDN のプロトコルの深い実装知識までは問われません。
> 「コントロールプレーンとデータプレーンの分離」「集中管理 vs 分散管理」
> 「南向き・北向きインターフェースの役割」を説明できることが重要です。

## 2. オーバーレイ・アンダーレイ・ファブリック

SDN やデータセンターネットワークを理解するうえで欠かせないのが、次の 3 つの用語です。

| 用語 | 意味 |
|---|---|
| **アンダーレイ（Underlay）** | 実際の物理ネットワーク。ルータ・スイッチ・ケーブル・IGP（内部ゲートウェイプロトコル）による IP 到達性など、転送の「土台」となる基盤 |
| **オーバーレイ（Overlay）** | アンダーレイの上に構築される論理（仮想）ネットワーク。トンネル技術を使い、物理構成に依存しない柔軟な接続を作る |
| **ファブリック（Fabric）** | アンダーレイ + オーバーレイ + コントローラを組み合わせた全体。単一の管理対象として扱えるネットワーク |

イメージとしては、アンダーレイが「道路網」、オーバーレイがその上を走る
「配送ルート（トンネル）」に相当します。物理的な道路がどうつながっていても、
オーバーレイのトンネルさえ通れば、離れた拠点やセグメント同士を 1 つの
ネットワークのように接続できます。

### VXLAN — 代表的なオーバーレイ技術

**VXLAN（Virtual Extensible LAN）** は、レイヤ 2 のフレームを UDP（宛先ポート
4789）でカプセル化し、IP（レイヤ 3）ネットワーク上を運ぶ技術です
（MAC-in-UDP）。これにより、物理的に離れたデータセンターや拠点間でも、
あたかも同じ L2 セグメントにいるかのように通信できます（L2 セグメントの延伸）。
データセンターのファブリックで広く使われる代表的なオーバーレイ技術です。

### Spine-Leaf とファブリック

Day 1 で学んだ **Spine-Leaf トポロジ**は、ファブリックのアンダーレイとして
よく使われます。すべての Leaf スイッチがすべての Spine スイッチに接続される
構成により、アンダーレイ上での安定した IP 到達性が確保され、その上に VXLAN
などのオーバーレイを重ねてファブリック全体を構築します。

```
        [Spine1]      [Spine2]      ← アンダーレイ（物理・IGPによるIP到達性）
        /  |  \        /  |  \
   [Leaf1][Leaf2][Leaf3][Leaf4]     ← Leafスイッチ

  Leaf1 ⇢⇢⇢ VXLANトンネル ⇢⇢⇢ Leaf4   ← オーバーレイ（論理）

  アンダーレイ + オーバーレイ + コントローラ = ファブリック
```

> **試験のポイント**: 「物理＝アンダーレイ／論理トンネル＝オーバーレイ」の区別は
> 頻出です。即答できるようにしておきましょう。

## 3. Cisco Catalyst Center（旧 DNA Center）

**Cisco Catalyst Center**（旧称 Cisco DNA Center）は、Cisco が提供する企業向けの
コントローラ／ネットワーク管理プラットフォームです。SDN コントローラの代表例の
1 つで、有線・無線を含むキャンパスネットワーク全体を集中管理します。

### 主なワークフロー

| 機能 | 内容 |
|---|---|
| **Design（設計）** | ネットワークの物理・論理構成（サイト階層、IP アドレス計画など）を事前に定義する |
| **Policy（ポリシー）** | セキュリティグループやアクセスポリシーを定義し、ネットワーク全体に適用する |
| **Provision（プロビジョニング）** | 設計・ポリシーに基づいて、実際の機器へ設定を展開する |
| **Assurance（監視・保証）** | 機器・クライアント・アプリケーションの健全性をダッシュボードで可視化し、問題の予兆を検知する |

### インテントベース・ネットワーキング（IBN）

**インテントベース・ネットワーキング（Intent-Based Networking: IBN）** とは、
管理者が「やりたいこと（意図・intent）」を高いレベルで入力すると、コントローラが
それを解釈して具体的な機器ごとの設定に変換し、展開する考え方です。管理者は
個々の機器の CLI コマンドを意識する必要がなくなります。

Catalyst Center は北向きに **REST API** を公開しており、外部の自動化スクリプトや
オーケストレーションツールから機器情報の取得や設定変更をプログラム的に行えます。
これは次節で学ぶ REST API の実務での使われ方の代表例です。

従来の「1 台ずつ CLI にログインして設定する」運用と比較すると、Catalyst Center
のような集中プロビジョニングとポリシー一括適用により、大規模ネットワークの
運用効率と一貫性が大きく向上します。

> **試験のポイント**: Catalyst Center の 4 つのワークフロー（Design / Policy /
> Provision / Assurance）と、IBN の考え方（意図 → 自動的に設定へ変換）は
> 頻出のキーワードです。

## 4. REST API の仕組み

**API（Application Programming Interface）** とは、ソフトウェア同士が機能や
データをやり取りするための取り決めです。中でも **REST（Representational State
Transfer）** は、HTTP をベースにした API のアーキテクチャスタイルで、コントローラや
クラウドサービスとのやり取りで最も広く使われています。REST API では、クライアントが
「リソース（機器・設定・情報など）」に対して操作を要求します。

### 主要な HTTP メソッドと CRUD 操作

REST API の HTTP メソッドは、データベース操作の基本である **CRUD**
（Create・Read・Update・Delete）に対応します。

| HTTP メソッド | CRUD 操作 | 意味 |
|---|---|---|
| **GET** | Read（読み取り） | リソースを取得する |
| **POST** | Create（作成） | 新規リソースを作成する |
| **PUT** | Update（置き換え） | リソース全体を置き換える |
| **PATCH** | Update（部分更新） | リソースの一部だけを更新する |
| **DELETE** | Delete（削除） | リソースを削除する |

### 主要なステータスコード

サーバはリクエストの結果を **HTTP ステータスコード**（3 桁の数値）で返します。
先頭の数字によって大まかな分類がされています。

| 分類 | 意味 |
|---|---|
| **2xx** | 成功 |
| **3xx** | リダイレクト |
| **4xx** | クライアント側のエラー |
| **5xx** | サーバ側のエラー |

代表的なコードは次のとおりです。

| コード | 名称 | 意味 |
|---|---|---|
| **200** | OK | リクエスト成功（GET など） |
| **201** | Created | リソースの新規作成に成功（POST など） |
| **400** | Bad Request | リクエストの構文や内容に誤りがある |
| **401** | Unauthorized | 認証情報が無い、または誤っている |
| **403** | Forbidden | 認証は正しいが権限が無い |
| **404** | Not Found | 指定したリソースが存在しない |
| **500** | Internal Server Error | サーバ内部でエラーが発生した |

> **試験のポイント**: HTTP メソッドと CRUD の対応、ステータスコードの意味
> （特に 200・201・401・404・500）と 2xx/4xx/5xx の分類は非常によく出題されます。
> 確実に覚えましょう。特に **401（認証エラー: あなたが誰か確認できない・未ログイン）**
> と **403（認可エラー: ログイン済みだが権限が無い）** は混同しやすいペアです。
> 「認証（401）→ 認可（403）の順」と覚えておきましょう。

### 認証方式

REST API では、誰がリクエストしているかを確認するために認証が必要です。

- **Basic 認証**: ユーザ名とパスワードをリクエストごとに送る
- **API キー**: 発行された固定のキーをヘッダなどに含めて送る
- **トークンベース認証**: 多くの場合、最初に POST でユーザ名・パスワードを送って
  トークンを取得し、以降のリクエストのヘッダにそのトークンを付与する
- **OAuth**: 第三者のアプリケーションに限定的な権限を委譲するための認証・認可の仕組み

### REST の特徴とリクエストの構成要素

REST には次のような特徴があります。

- **ステートレス**: 各リクエストは独立しており、サーバは前回のリクエストの状態を
  覚えていない（必要な情報は毎回リクエストに含める）
- **リソースは URI で指定**: 操作対象は URI（例: `/api/v1/devices/1`）で表される
- **データ表現は主に JSON**（XML が使われることもある）

リクエストは主に次の要素から構成されます。

- **メソッド**: GET / POST / PUT / PATCH / DELETE
- **URI**: 操作対象のリソースを示すアドレス
- **ヘッダ**: `Content-Type: application/json`（送信データの形式）、
  `Authorization: Bearer <token>`（認証情報）など
- **ボディ**: 送信するデータ本体（主に JSON。GET では通常不要）

> 💼 **実務では**: Catalyst Center などの API を触る前に、Postman や curl で
> エンドポイントを試打ちして挙動を確かめてから自動化スクリプトに組み込むのが
> 定石です。トークンには有効期限（Catalyst Center の X-Auth-Token は既定で
> 約 60 分）があり、期限切れで 401 が返るのは新人が最初に必ずハマるポイントです。
> また、認証トークンやパスワードをスクリプトに直書きして Git にコミットし
> 漏洩させる事故が後を絶たないため、実務では環境変数やシークレット管理
> （Vault 等）に逃がすのが鉄則です。

## 5. JSON の読み書き

**JSON（JavaScript Object Notation）** は、軽量なデータ交換フォーマットで、
REST API のリクエスト・レスポンスのボディとして最も広く使われています。

### データ型

| 型 | 例 | 備考 |
|---|---|---|
| 文字列（string） | `"Router1"` | 二重引用符 `""` で囲む |
| 数値（number） | `24`, `3.5` | 引用符で囲まない |
| 真偽値（boolean） | `true`, `false` | 引用符で囲まない |
| null | `null` | 値が無いことを表す |
| オブジェクト（object） | `{ "key": "value" }` | `{}` で囲む。キー:値のペアの集合 |
| 配列（array） | `[1, 2, 3]` | `[]` で囲む。値を順序付きで並べる |

### JSON のサンプル

```json
{
  "hostname": "SW1",
  "managementIp": "192.168.10.1",
  "interfaces": [
    { "name": "GigabitEthernet0/0", "status": "up" },
    { "name": "GigabitEthernet0/1", "status": "down" }
  ],
  "isActive": true
}
```

上の例では、`interfaces` キーの値が「オブジェクトの配列」になっています。
1 番目のインターフェース（配列のインデックス 0）の `status` の値は `"up"` です。
このように、オブジェクトや配列を組み合わせてネスト（入れ子）にすることができます。

### 構文ルール

- オブジェクトの**キーは必ず二重引用符 `""` で囲む**（シングルクオート不可）
- キー:値のペアは**コロン `:`** で区切る
- 複数のペアや配列の要素は**カンマ `,`** で区切るが、**末尾の要素の後にカンマは付けない**
  （末尾カンマは構文エラーになる）
- オブジェクト `{}` や配列 `[]` は何重にもネストできる

> **試験のポイント**: JSON のスニペットを見せて、特定のキーの値を答えさせる問題や、
> 構文の正誤（末尾カンマ、引用符の有無など）を判定させる問題が出題されます。

### JSON・XML・YAML の違い

| 形式 | 特徴 |
|---|---|
| **JSON** | `{}` `[]` で構造を表す。REST API で最も一般的 |
| **XML** | タグ（`<tag>...</tag>`）で構造を表す。冗長だが古くから広く使われる |
| **YAML** | インデント（字下げ）で構造を表す。可読性が高く、**Ansible の Playbook** で採用されている |

## 6. 構成管理ツール（Ansible / Terraform）

複数の機器を手作業で 1 台ずつ設定していると、意図しない設定の差異（**設定ドリフト**）
が発生しやすくなります。**構成管理ツール**は、あるべき設定内容をコード（テキスト）
として定義し、それを多数の機器へ自動的に適用することで、設定を標準化しドリフトを
防ぎます。

### Ansible

- **エージェントレス**: 管理対象機器に専用ソフトウェアをインストールする必要がない
- **SSH ベース**で動作する（ネットワーク機器へは主に SSH 経由で接続）
- **YAML** 形式の **Playbook** に「あるべき状態」を記述する
- **Push モデル**: コントロールノード（Ansible を実行する管理端末）から対象機器へ
  能動的に設定を配布する

### Puppet / Chef（比較知識）

- **エージェントベース**: 対象機器に常駐エージェントをインストールする必要がある
- 独自の **DSL（ドメイン固有言語）** や **Ruby** ベースの記法を使う
- 基本的に **Pull モデル**: エージェントが定期的にサーバへ設定を取りに行く

### Terraform

- **HashiCorp** 社が開発
- **HCL（HashiCorp Configuration Language）** という言語を使う
- **インフラのプロビジョニング（IaC: Infrastructure as Code）** に強みを持つ
- **宣言的**: 「最終的にどうあるべきか」を記述すると、その状態を維持するように動作する

### ツール比較表

| ツール | エージェント | モデル | 言語 | 主な用途 |
|---|---|---|---|---|
| Ansible | エージェントレス | Push | YAML | 構成管理・自動化 |
| Puppet | エージェントベース | Pull | 独自 DSL | 構成管理 |
| Chef | エージェントベース | Pull | Ruby ベース | 構成管理 |
| Terraform | エージェントレス | 宣言的（適用時に反映） | HCL | インフラのプロビジョニング（IaC） |

### 共通の利点

- **冪等性（べきとうせい）**: 同じ処理を何度実行しても結果が変わらない
  （すでに望ましい状態なら何もしない）
- **バージョン管理**: 設定コードを Git などで履歴管理できる
- **再現性**: 同じコードを使えば誰でも同じ環境を再現できる
- **大規模展開の効率化**: 手作業に比べて圧倒的に速く、ミスが少ない

> 💼 **実務では**: Playbook や Terraform コードは必ず Git に置き、PR レビューを
> 通してから本番へ適用します。Ansible なら本適用の前に必ず `--check`
> （ドライラン）で差分を確認するのが基本で、新人がやりがちな事故は「検証用の
> つもりが本番インベントリに向けて実行してしまう」ことです。冪等性のおかげで
> 再実行は安全ですが、それは適用先が正しい場合の話です。Terraform では state
> ファイルの共有・保護（リモート state）を怠るとチーム作業で構成が壊れるため、
> 最初に整備すべき勘所です。

> **試験のポイント**: 「エージェントの有無」「Push か Pull か」「使用言語」の
> 組み合わせで各ツールを識別できるようにしておきましょう。特に Ansible の
> 「エージェントレス・YAML・Push・SSH」は頻出です。

## 7. ネットワーク運用における AI の影響（v1.1 追加）

CCNA 200-301 は 2024 年 8 月の **v1.1 改訂**で、AI（人工知能）のネットワーク運用への
影響が新たに試験範囲に追加されました。AI の中核技術となっているのが
**機械学習（Machine Learning: ML）** で、過去の大量のデータからパターンを
学習します。**予測 AI** は主にこの ML に基づいて将来の事象を予測し、
**生成 AI** は学習した内容をもとに新しいコンテンツを生成します。ここでは特に
**生成 AI** と **予測 AI** の違いを押さえます。

| 種類 | 内容 | ネットワーク運用での活用例 |
|---|---|---|
| **生成 AI（Generative AI）** | 大量のデータを学習し、新しいコンテンツ（文章・設定案・スクリプト・要約など）を**生成**する | 設定案のドラフト作成、トラブルシュートの手順案内、ドキュメントの自動要約 |
| **予測 AI（Predictive AI）** | 過去のデータのパターンから将来の事象を**予測**する | 障害の予兆検知、トラフィック量の予測、キャパシティ計画 |

### ネットワーク運用への効果

- **異常検知の高速化**: 通常のパターンから外れた挙動を素早く検知する
- **トラブルシューティング支援**: 過去の類似事例や設定を参照した対応案の提示
- **運用の自動化・省力化**: 定型的な作業や監視業務の負荷を軽減する

### 留意点

- AI の出力は必ず**人間が検証する必要がある**（誤り・幻覚〈事実に基づかない
  もっともらしい出力〉のリスクがあるため）
- 学習データや運用データの取り扱いには**データプライバシーとセキュリティ**への
  配慮が欠かせない

> **試験のポイント**: CCNA では AI の実装の詳細ではなく、**生成 AI と予測 AI の
> 違い**、機械学習（ML）が両者の土台になっている点、およびネットワーク運用への
> 概念レベルでの利点（異常検知・トラブルシュート支援・自動化）が問われます。

## 8. まとめ

- 従来型ネットワークは分散管理、SDN はコントロールプレーンを集中管理し、
  SBI（機器制御）と NBI（アプリ連携）でやり取りする
- アンダーレイ（物理）・オーバーレイ（論理トンネル、例: VXLAN）・ファブリック
  （両者 + コントローラの全体）を区別する
- Cisco Catalyst Center は Design/Policy/Provision/Assurance の 4 機能を持つ
  コントローラで、IBN の考え方を体現する
- REST API は HTTP メソッド（GET/POST/PUT/PATCH/DELETE = CRUD）とステータス
  コード（2xx〜5xx）で成否を表現する
- JSON はオブジェクト `{}` と配列 `[]` を組み合わせてデータを表現する軽量フォーマット
- Ansible（エージェントレス・YAML・Push）、Terraform（HCL・宣言的・IaC）など
  構成管理ツールは設定ドリフトを防ぎ冪等性を実現する
- 生成 AI（コンテンツ生成）と予測 AI（将来予測）はネットワーク運用の効率化に
  貢献するが、出力は人間の検証が必要

## 確認問題（自己チェック・解答は末尾）

1. SDN において、経路を「決める」機能と「転送する」機能をそれぞれ何と呼ぶか。
2. コントローラが配下の機器を制御するために使うインターフェースの名称と、
   アプリケーションと連携するために使うインターフェースの名称をそれぞれ答えよ。
3. VXLAN はアンダーレイ・オーバーレイのどちらに分類される技術か、理由とともに答えよ。
4. 新規リソースを作成する REST API の HTTP メソッドと、そのとき成功時に
   返される代表的なステータスコードを答えよ。
5. Ansible の特徴を「エージェントの有無」「モデル（Push/Pull）」「使用言語」の
   観点で説明せよ。

<details><summary>解答</summary>

1. 決める機能: コントロールプレーン／転送する機能: データプレーン
2. 配下の機器を制御: 南向きインターフェース（SBI）／アプリと連携: 北向き
   インターフェース（NBI）
3. オーバーレイ。物理ネットワーク（アンダーレイ）の上に、UDP を使ったトンネルで
   L2 フレームをカプセル化して論理的な接続を作るため
4. メソッド: POST／ステータスコード: 201 Created
5. エージェントレス（対象機器にソフトウェア不要）、Push モデル（コントロール
   ノードから能動的に配布）、YAML で Playbook を記述する
