# 受講者ペルソナ（教材・運用の判断基準）

教材の書き方・コラムの内容・講師の対応で迷ったら、このペルソナに立ち返る。
（2026-07 ユーザー確認済み）

## 前提（確定事項）

- **受講者の中心（数的メイン）はペルソナ1（完全未経験）**。教材の難易度・
  言葉選び・手順の粒度は佐藤さんを基準にする
- **研修後の実務は SES（客先常駐）の保守・運用がメインターゲット**。
  監視・アラート対応・手順書ベースの定型作業・切り分け・エスカレーション・
  作業報告が最初の仕事になる
- **人物評価によっては構築案件へステップアップできるとベター**。
  「保守で信頼を積む → 検証・ドキュメント・報告の質で評価される → 構築へ」
  というキャリアの階段を研修中から意識させる

## ペルソナ1：佐藤さん（23歳）— 完全未経験【受講者の中心】

- 文系卒 → 営業1年 → 未経験でIT転職。配属前研修として受講
- スマホは使いこなすがPCはレポート程度。タイピングは人差し指。2進数は初耳
- **山場**: Week0 P2（2進数）と Day3（サブネット計算）、CLIへの心理的ハードル。
  エラーが出ると「壊したかも」と手が止まる
- **支える仕組み**: Week0全体、毎朝のタイピング10分、30分ルール（質問文化）、
  誤答ノート
- **卒業後の姿**: 客先の運用保守チームに配属。show コマンドでの状態確認、
  手順書に沿った作業、先輩へのエスカレーションから始める

## ペルソナ2：田中さん（34歳）— 異業種からのキャリアチェンジ

- 販売職10年。Excelは使えるがネットワーク知識ゼロ。家庭があり本気度が高い
- **山場**: 知識より時間との戦い（7h×25日の集中と両立）。計算への苦手意識
- **支える仕組み**: 誤答ノート×分散復習（時間効率）、週次振り返りでの負荷調整、
  「合格まで研修内で伴走」という出口の明確さ
- **卒業後の姿**: 社会人力（報告・調整）が武器。保守現場で顧客との
  コミュニケーションを評価され、構築へのステップアップ候補になりやすい

## ペルソナ3：鈴木さん（27歳）— 独学でかじった半歩経験者【少数派】

- 社内ヘルプデスク2年 + 自作PC趣味。知識は断片的で「なぜ」が説明できない
- Week0スキップ判定（--skip-precourse）の対象になり得る
- **山場**: 「わかったつもり」。小テストは取れるがラボで手が動かない、
  自己評価と実点数の乖離
- **支える仕組み**: 講師ガイドの「理解の錯覚の検出」、口頭確認、ラボの完了条件

## 設計への含意

1. **「実務では」コラムの軸は保守・運用現場**にする:
   - show / debug コマンドは「監視と切り分けの道具」として紹介する
   - 手順書文化（作業前確認・戻し手順・作業報告）、エスカレーションの作法、
     「勝手に設定変更しない」現場ルールに触れる
   - 構築の話題は「保守で信頼を積んだ先のステップ」として位置づける
2. **評価される力を研修で鍛える**: ラボの提出物（検証結果の記録・スクリーンショット・
   わかったことの言語化）は、そのまま保守現場の作業報告の練習である。
   講師はこの観点でも提出物をフィードバックする
3. **難所の設計**: 佐藤さん基準で Week0 P2 と Day3 に最厚のサポート
   （講師の重点フォロー日として運用ガイドに明記）
