# Backlog プロジェクト構成設計

CCNA 研修を Backlog 上で運用するためのプロジェクト設計です。

## 0. マージ先テンプレート（確定）

投入先は、ユーザー提供のプロジェクトテンプレート
**「インフラ開発研修（CCNA）のテンプレート」**（`backlog/project-template.json`）
から作成されるプロジェクト。テンプレートの設定と本設計の適合:

| テンプレート設定 | 本設計との適合 |
|---|---|
| 書式 = **Markdown** | ✅ 教材・課題テンプレート・Wiki投入（画像の `![alt](添付名)` 参照含む）がそのまま使える |
| ドキュメント有効 / Wiki有効（ツリー表示） | ✅ 「Wiki API投入 → ドキュメント移行」の想定どおり |
| 状態 = 未対応/処理中/処理済み/完了 | ✅ 本設計の状態運用と一致 |
| 子課題あり（孫課題なし） | ✅ 受講者別プロジェクト方式なら子課題は補助的に使用 |
| ファイル共有有効 / Git無効 | ✅ .pkt 添付は課題添付とファイル共有で対応 |
| 種別 = 基本操作/プロジェクト設定/要望/その他 | ⚠️ 研修用の種別（講義/ラボ/小テスト/週次テスト/模試）は投入スクリプトが**追加**する。テンプレート由来の4種別は残るので、不要なら手動整理するか、テンプレート側に研修用種別を追加しておくと新規プロジェクトが最初から揃う（推奨） |
| カテゴリー/マイルストーン/カスタム属性 = 空 | ✅ 投入スクリプトが作成する |

## 1. プロジェクト基本設定

| 項目 | 設定 |
|---|---|
| プロジェクト名 | CCNA 研修 2026 |
| プロジェクトキー | `CCNA` |
| チャート | ガントチャート・バーンダウンを有効化（進捗の可視化に使用） |
| 参加者 | 受講者 = 一般ユーザー、講師 / メンター = プロジェクト管理者 |

## 2. 種別（課題タイプ）

| 種別 | 用途 | 色の目安 |
|---|---|---|
| `講義` | その日のドキュメント学習。教材ドキュメントへのリンクと学習目標を記載 | 緑 |
| `ラボ` | Packet Tracer 演習。手順書リンクと `.pka` / `.pkt` ファイルを添付 | 青 |
| `小テスト` | 日次 10 問。設問は課題本文、解答はコメントで提出 | 赤 |
| `週次テスト` | Day 5/10/15/20 の 25 問（修了テストは 60 問） | 橙 |

## 3. カテゴリー（= 試験ドメイン）

1. `1-ネットワーク基礎`
2. `2-ネットワークアクセス`
3. `3-IPコネクティビティ`
4. `4-IPサービス`
5. `5-セキュリティ基礎`
6. `6-自動化とプログラマビリティ`

課題に付与しておくと「どのドメインでつまずいているか」をカテゴリー別の絞り込みで分析できる。

## 4. マイルストーン（発生バージョン）

| マイルストーン | 期間 | 内容 |
|---|---|---|
| `Week1-ネットワーク基礎` | Day 1〜5 | ドメイン 1 |
| `Week2-ネットワークアクセス` | Day 6〜10 | ドメイン 2 |
| `Week3-IPコネクティビティとIPサービス` | Day 11〜15 | ドメイン 3・4 |
| `Week4-セキュリティと自動化` | Day 16〜20 | ドメイン 5・6 + 総まとめ |

週次テスト合格（70% 以上）をもってマイルストーン完了とする。

## 5. 課題の構成

Week0（5 日 × 3 = 15）+ 本編（20 日 × 3 = 60）+ 試験対策フェーズ（8）で
**計 83 課題**（`--skip-precourse` 時は Week0 を除き Day00 を追加して 69 課題）。命名規則:

```
[Day01] 講義: ネットワークの全体像と OSI / TCP-IP モデル
[Day01] ラボ: PT入門 — 最小構成の作成とカプセル化の観察
[Day01] 小テスト: OSI モデルとカプセル化
```

- **担当者**: 受講者本人（複数受講者の場合は受講者ごとに課題を複製するか、
  子課題で受講者別に分ける）
- **期限日**: 研修開始日から営業日ベースで自動計算（投入スクリプトが設定）
- **状態運用**: 未対応 → 処理中（着手）→ 処理済み（提出）→ 完了（講師確認後）

課題の一括投入は [`scripts/create-backlog-issues.mjs`](./scripts/create-backlog-issues.mjs) を使用。

### 課題テンプレート（種別ごとに登録）

**講義:**

```
## 学習目標
- （このセッションで説明できるようになること）

## 教材
- ドキュメント: （教材ドキュメントへのリンク）

## 進め方
1. 教材ドキュメントを通読する（3.5h 目安）
2. 理解チェック（教材末尾の確認問題）を解く
3. 疑問点はこの課題のコメントに書き残す → 講師が回答

## 完了条件
- [ ] 教材を通読した
- [ ] 確認問題を解いた
- [ ] 疑問点をコメントした（なければ「なし」とコメント）
```

**ラボ:**

```
## ゴール
- （構築するトポロジと動作要件）

## 手順書
- ドキュメント: （ラボ手順書へのリンク）
- 添付: 開始用ファイル（.pkt / .pka）

## 提出物
- 完成した .pkt ファイルをこの課題に添付
- 検証コマンドの実行結果（スクリーンショットまたはテキスト）をコメント

## 完了条件
- [ ] 要件どおりの疎通 / 動作を確認した
- [ ] 完成ファイルを添付した
```

**小テスト:**

```
## ルール
- 制限時間 30 分・教材参照なしで解くこと
- 解答はこの課題の**コメント**に「Q1: A」の形式で提出
- 正答率 70% 未満の場合は翌朝に再テスト

## 設問
（10 問をここに記載。選択式中心 + 記述 1〜2 問）

## 解答・解説
- 翌日 9:00 に「解答解説」ドキュメントのリンクをコメントで公開
```

## 6. ドキュメント機能のフォルダ構成

```
📁 00_ガイダンス
   ├─ 研修の進め方（このプロジェクトの使い方）
   ├─ カリキュラム全体表
   └─ Packet Tracer 導入マニュアル
📁 01_教材
   ├─ 📁 Week1_ネットワーク基礎
   │    ├─ Day01_ネットワークの全体像とOSIモデル
   │    ├─ Day02_Cisco IOS の基本操作
   │    └─ ...
   ├─ 📁 Week2_ネットワークアクセス
   ├─ 📁 Week3_IPコネクティビティとIPサービス
   └─ 📁 Week4_セキュリティと自動化
📁 02_ラボ手順書
   ├─ 📁 Week1 … Day01_PT入門ラボ 手順書 など
   └─ ...
📁 03_小テスト解答解説
   ├─ Day01_解答解説（翌日公開）
   └─ ...
📁 04_講師用（権限で受講者から隠す）
   ├─ 採点基準
   └─ 週次テスト・修了テスト原本
```

- 解答解説と講師用フォルダは**公開タイミング / 権限に注意**（プロジェクト分離で
  構造的に隠すのが堅い。詳細は §6.5）。
- **投入方法**: ドキュメント API は未提供のため、(a) Markdown 教材を Wiki API で
  一括投入 → Wiki 移行機能でドキュメント化、または (b) 手動貼り付け。
  具体的な操作手順は **§6.5**。

## 6.5 投入と公開の手順書（Wiki 投入 → ドキュメント移行 → 権限設定）

教材を Backlog に載せて受講者へ公開するまでの実作業手順です。①③④は Claude（スクリプト）で、
**②の「Wiki → ドキュメント移行」は Backlog 画面での手作業**になります（ドキュメント機能に
API がないため）。

> ⚠️ **画面メニュー名について**: Backlog の UI はバージョンで表記が変わります。以下の
> メニュー名は目安です。実画面で近い名称の導線を探し、見当たらない場合は末尾の
> 「代替: 手動貼り付け」に切り替えてください。

### 前提となるプロジェクトの分け方（モデル A / 09 の権限原則に対応）

**フォルダ単位の権限に頼らず、プロジェクトのメンバーシップで構造的に隠す**のが安全です。

| プロジェクト | 置くもの | 受講者メンバー | 投入方法 |
|---|---|---|---|
| **CCNA-教材**（共有・全受講者が閲覧） | `00_ガイダンス` / `01_教材` / `02_ラボ手順書`（＝見せてよいもの） | ○（閲覧） | `upload-wiki.mjs`（**quiz を含めない**） |
| **CCNA-講師用**（受講者を入れない） | 解答解説 / 模試設問 / 採点基準 / テスト原本 | **×** | `upload-wiki.mjs --include-quiz` ＋手動 |
| CCNA-〈氏名〉（受講者別） | その受講者の課題・採点コメント・振り返り | ○（本人のみ） | `create-backlog-issues.mjs` |

### 手順 A：公開教材を CCNA-教材へ

1. **前提確認**: プロジェクトの書式 = Markdown、Wiki 有効・ドキュメント有効。
   API キーのユーザーがそのプロジェクトの**管理者**であること。
2. **内容確認（dry-run）**: ネットワーク許可済み環境で
   `node scripts/upload-wiki.mjs --project <教材KEY> --dry-run`
   （**`--include-quiz` は付けない** ＝ 解答は投入しない）。
3. **本投入**: `BACKLOG_SPACE_URL` と `BACKLOG_API_KEY` を設定し、`--dry-run` を外して実行。
   → プロジェクト > **Wiki** に `CCNA研修/00_ガイダンス/…` のツリーで入る
   （ページ名の `/` が階層になる。冪等なので再実行で更新される）。
4. **目視確認**: Wiki ツリーで階層・見出し・画像（課題添付として表示）を確認。
5. **Wiki → ドキュメント移行（手作業）**: ドキュメント画面（または Wiki 画面）の
   「**…（メニュー）> ドキュメントへ移行 / Wiki から取り込み**」に相当する導線を実行。
   → ドキュメントに `00_ガイダンス`〜`02_ラボ手順書` のフォルダができる。
6. **整理**: フォルダの並び順・不要な階層を調整（一度きり）。

### 手順 B：非公開コンテンツを CCNA-講師用へ

7. 受講者を入れない **CCNA-講師用**プロジェクトに対し、
   `node scripts/upload-wiki.mjs --project <講師用KEY> --include-quiz` で
   解答解説（quiz）を Wiki 投入 → 手順 5 と同様に移行。
8. 模試設問・採点基準・テスト原本は、Markdown があれば同様に、なければ手動で
   ドキュメント作成（`04_講師用`）。

### 手順 C：権限（受講者から隠す）

9. **受講者は CCNA-教材と自分の CCNA-〈氏名〉のみメンバーにする**。CCNA-講師用には**入れない**。
   これで解答・模試・講師用は構造的に見えない（§09 の原則）。
10. 解答解説の「翌日公開」（§7）は、講師が該当解説を**受講者の個別プロジェクトのコメント
    またはドキュメント**に貼って開示する（教材プロジェクトには置かない）。

### 代替: 手動貼り付け（移行機能が見つからない / 少量のとき）

- ドキュメントで対象フォルダを作り、**新規ドキュメント**に各 Markdown を貼り付ける。
  書式が Markdown なら見出し・表・コードブロックはそのまま反映される。
- 画像は本文の `![alt](ファイル名)` に対応する画像を、そのドキュメントに**添付**する。

### うまくいかないとき

- **Wiki ツリーが平坦になる**: ページ名の `/` 区切り（`CCNA研修/00_ガイダンス/…`）が
  保たれているか確認。
- **画像が表示されない**: 添付ファイル名と本文の `![alt](名前)` が一致しているか確認。
- **移行後に受講者へ解答が見えてしまう**: それは CCNA-教材側に quiz/解答を入れた兆候。
  手順 A-2 で `--include-quiz` を付けていないか、投入先プロジェクトを取り違えていないか確認。

## 7. 小テストの運用フロー

1. 小テスト課題は投入時から本文に設問を記載（期限日 = 当日）
2. 受講者はコメントで解答を提出し、状態を「処理済み」に変更
3. 講師（または自己採点）が採点し、点数をコメント → 状態「完了」
4. 翌朝、解答解説ドキュメントのリンクをコメントで公開
5. 70% 未満の受講者には「再テスト」課題を追加（テンプレートから作成）

> 将来の拡張: Backlog API の Webhook + 採点 Bot（コメントの解答をパースして
> 自動採点・自動コメント）を作れば採点を自動化できる。まずは手動運用で開始し、
> 受講者数が増えたら Bot 化を検討する。

## 8. 進捗の可視化

- **ボード**: 状態別（未対応 / 処理中 / 処理済み / 完了）で当日のタスクを確認
- **ガントチャート**: 20 日間の全体進捗と遅れの検知
- **カテゴリー絞り込み**: ドメイン別の得意 / 不得意の把握
- **週次テスト結果**: マイルストーンの完了状況で判定
