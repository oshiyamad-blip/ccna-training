# 講師用運用ガイド

> 配置先: ドキュメント `04_講師用 > 運用ガイド`（**受講者には非公開**の
> フォルダに置き、権限設定で閲覧を制限すること）

> **ローリング型運用（受講者が随時入学・卒業する場合）**: プロジェクト構成・
> 入学 / 卒業手順・週次運用の読み替え・負荷の目安は
> `06-rolling-operations.md`（ローリング型運用ガイド）を先に読んでください。
> 本ガイドの「金曜」「翌週月曜」等の曜日表記は、ローリング型では
> 「各受講者の Day 5 / 10 / 15 / 20」「その翌営業日」と読み替えます。

## 1. 開講前の準備チェックリスト（開講 1 週間前まで）

- [ ] Backlog プロジェクト作成（キー例: `CCNA`）。受講者を一般ユーザー、
      講師をプロジェクト管理者として招待
- [ ] 課題の一括投入: `scripts/create-backlog-issues.mjs` を実行
      （`--start` に Day 1 の日付、受講者が 1 名なら `--assignee` も指定。
      複数名の場合は受講者ごとに実行するか、投入後に子課題で分ける）
- [ ] 手動で課題 `[Day00] 環境構築` を追加（期限 = 開講前日。内容 =
      Packet Tracer 導入マニュアルのチェックリスト）
- [ ] ドキュメントのフォルダ作成（`00_ガイダンス` / `01_教材` / `02_ラボ手順書` /
      `03_小テスト解答解説` / `04_講師用` / `05_誤答ノート` / `06_振り返り`）と
      権限設定（`04_講師用` は講師のみ）
- [ ] 教材の投入: `materials/` の Markdown を Wiki API で一括投入
      （`scripts/upload-wiki.mjs`）→ Backlog の Wiki 移行機能でドキュメント化、
      または手動でドキュメントへ貼り付け。
      **注意: `dayNN-quiz.md` には解答・解説が含まれる**。既定でアップロード対象外に
      なっているので、設問部分だけを毎日、小テスト課題の本文に貼る運用とする
- [ ] 課題テンプレート 3 種（講義 / ラボ / 小テスト）を登録（`02-backlog-design.md` 参照）
- [ ] ラボの開始用 `.pkt` ファイルを作成（`materials/pkt-build-spec.md` に従う。初回のみ）し各ラボ課題に添付（各ラボ手順書のトポロジを
      Packet Tracer で組んで保存したもの。初回開講時に作成すれば次回から流用可能）

## 1.5 講師体制（兼務 2 名での役割分担）

本研修は**兼務講師 2 名**で運用します。負荷を分散しつつ属人化を防ぐため、
次の分担を推奨します。

| 役割 | 内容 | 交代 |
|---|---|---|
| **当番講師**（日替わり） | その日の質問一次対応（1 時間以内の返信）、朝の解答解説公開、夕方の採点確定 | 1 日交代（例: 偶数日 A / 奇数日 B） |
| **担当講師**（受講者ごとに固定） | 週次テスト後の個別レビュー 15 分、振り返りの Try 確認、遅延時の個別フォロー | 受講者の入学時に割り当て、卒業まで固定 |

- 採点の一次処理は**ツールに任せる**: 選択式は `grade-quiz.mjs`（機械採点・確実）、
  記述式を含む場合は `ai-grade.mjs`（AI 一次採点 + フィードバック文生成）。
  当番講師は結果を確認して確定するだけにする（1 名あたり 5 分以内が目安）
- 技術的に自信のない質問は無理に即答せず「担当外なので◯◯さんから回答します」と
  つないで OK。回答の質 > 速さ
- 週 1 回 15 分、講師 2 名で「詰まっている受講者・教材の改善点」を共有する

### AI 一次採点の使い方（記述式を含む小テスト）

```bash
cd scripts && npm install   # 初回のみ
BACKLOG_SPACE_URL=... BACKLOG_API_KEY=... ANTHROPIC_API_KEY=... \
node ai-grade.mjs --issue CCNA-42 --day 3 --post
```

- 教材の解答・採点基準を正解ソースとして、選択式＋記述式を採点し、
  受講者向けフィードバック文まで生成して課題にコメントする
- コメントには「AI一次採点・講師確認前」と明記される。**講師は判定
  （特に「要講師確認」と出た設問）を確認してから状態を「完了」にする**
- AI の判定と教材に食い違いを見つけたら、教材側の解説を改訂する（Git が原本）

## 1.6 初心者対応の心得（受講者は IT リテラシーゼロ前提）

- **Week 0 スキップ判定**: 口頭確認 3 点 + **実技 2 問**で判定する
  （シミュレーションで「知識テストだけでは概念の欠損を検出できない」ことが
  確認されたため）。
  ①2 進数⇔10 進数の変換ができる ②IP アドレスとは何か説明できる
  ③ファイル操作・スクリーンショットに迷いがない
  ④【実技】VLSM の小問 1 問を紙で解ける ⑤【実技】Packet Tracer で
  PC2 台 + スイッチの ping 疎通を 15 分以内に組める。
  5 点すべて揃った場合のみ省略可（投入スクリプトの --skip-precourse）
- **用語を使ったら定義する**: 質問への回答・コメントでも、教材と同じく専門用語を
  裸で使わない。「受講者が知らない言葉で回答する」のが初心者離脱の最大要因
- **質問が出ないことを危険信号として扱う**: 初心者は「何がわからないかが
  わからない」状態に陥る。1 日質問ゼロの受講者には講師側から「今日いちばん
  ピンとこなかった箇所はどこか」を具体的に聞く
- **重点フォロー日**: Week0 P2（2 進数）と Day 3（サブネット計算）は全ペルソナが
  つまずく最大の山。この 2 日は当番講師が進捗をこまめに見る
- **「わかったつもり」の検出**: 小テスト（選択式）は再認で通過できてしまう。
  ラボの完了条件と口頭確認を一次検出線とし、自己評価と実点数の乖離が出た
  受講者を優先フォローする
- **教材の説明不足は教材の改訂で返す**: 同じ質問が 2 回出たら教材（Git）を直す

## 2. 日次運用ルーチン

### 朝（9:00）

1. 前日の小テストの**解答・解説を公開**する:
   `materials/weekN/dayNN-quiz.md` の「解答・解説」部分を
   `03_小テスト解答解説 > DayNN` ドキュメントとして作成し、
   前日の小テスト課題にリンクをコメント
2. 再テスト対象者（70 点未満）がいれば「再テスト」課題を作成（テンプレートから）
3. 受講者が朝の振り返り（誤答ノートの解き直し → 講義冒頭のウォームアップ
   想起クイズ）を行っていることを確認する。誤答ノートが 2 日以上更新されて
   いない受講者には個別に声をかける（小テストで間違いがあるのにノートが
   空なのは要注意サイン）

### 日中

- 講義・ラボ課題のコメント（質問）に随時回答する。
  目安: **質問から 1 時間以内に一次回答**
- ラボの「処理済み」課題を確認し、提出物（.pkt・検証結果）をレビューして
  「完了」に変更。要件を満たさない場合は差し戻しコメント

### 夕方（17:30 以降）

1. 当日の小テストを採点する。手動採点、または採点支援スクリプト:

   ```bash
   BACKLOG_SPACE_URL=... BACKLOG_API_KEY=... \
   node scripts/grade-quiz.mjs --issue CCNA-42 --answers "B,C,C,B,C,C,C,B,C" --post
   ```

   （記述問題は自動採点対象外。スクリプトの出力を見て手動で加点し、
   合計点をコメントする）
2. 点数と再テスト要否を課題コメントに記入し、状態を「完了」へ
3. 進捗の確認: ボードで当日 3 課題が完了しているか。遅れている受講者には
   個別フォロー（翌日の計画をコメントで合意）

## 3. 採点基準

| テスト | 配点 | 合格 | 不合格時 |
|---|---|---|---|
| 日次小テスト（10 問） | 各 10 点 | 70 点 | 翌朝 9:30 まで再テスト（別問題 or 同問題の口頭確認） |
| 週次テスト（25 問） | 各 4 点 | 70 点 | 週末補講 + 再テスト。合格でマイルストーン完了 |
| 修了テスト（60 問） | ドメイン比率配点 | 75% | 弱点ドメインの補講 + 3 営業日以内に再テスト |

- 記述問題: 趣旨が合っていれば満点、一部なら半分。解答例は各 quiz ファイルの
  解説表を基準とする
- ラボ: 点数化しない。「要件充足 / 差し戻し」の 2 値。差し戻しは翌日午前まで

## 4. 週次運用

- 金曜夕方: 週次テスト採点 → 合格者のマイルストーンを「完了」に
- テスト後 30 分で受講者に**週次振り返り**（Keep / Problem / Try。書式は
  `00_ガイダンス > 週次振り返りの書き方`）を書いてもらう。
  「講師への質問」欄には翌営業日までに回答し、「Try」は翌週の振り返りで
  実行されたか一緒に確認する
- 週次テストの**誤答をドメイン別に集計**し、受講者の自己評価（振り返りの
  ドメイン別手応え）と突き合わせて、翌週月曜朝に 15 分の弱点解説を行う
  （Day 20 の総復習でもこの集計を使う）。自己評価と実点数が乖離している
  受講者（「◎なのに誤答が多い」）は理解の錯覚が起きているため優先フォロー
- ガントチャートで全体進捗を確認。2 日以上の遅延があれば計画を再調整
  （ラボの一部を宿題化する等。講義と小テストは削らない）

## 5. 修了判定と試験対策フェーズ

- 修了条件: 修了テスト 75% 以上 **かつ** Day 20 総合演習ラボの要件充足 →
  **試験対策フェーズ（Day 21〜）へ進む**。自習期間は設けない
- Day 21〜25 は模試サイクル（模試 → 誤答分析 → 弱点補強）。受験 GO 判定・
  日次レビュー・受験予約の運用は `07-exam-phase.md` に従う
- **CCNA 合格をもって卒業**。受験結果（合否・スコア・ドメイン別）を必ず記録し、
  教材・模試の改訂に反映する
- 研修の振り返り: 全課題のコメントを見返し、質問が集中したトピックを
  次回開講時の教材改訂に反映する（`materials/` を更新）

## 6. よくある運用トラブル

| 状況 | 対応 |
|---|---|
| 受講者が小テストで教材を見ている疑い | 点数と理解度の乖離をラボと口頭確認で検出。詰問せず「口頭で説明してもらう」運用に切り替える |
| ラボが時間内に終わらない受講者が多い | 開始用 .pkt の完成度を上げる（配線済みにする等）。手順の時間目安を見直す |
| 質問が出ない | 「30 分ルール」を再周知。講師から「どこまで進んだ？」と課題コメントで先に声をかける |
| Packet Tracer の不具合 | `03-packet-tracer-manual.md` のトラブルシューティング表 → 解決しなければ PT の再起動・ファイル再配布 |
