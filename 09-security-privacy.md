# セキュリティ・プライバシー運用ガイド

CCNA 研修は受講者の個人データが継続的に集まる仕組みです。本ガイドは、集まる
データの棚卸し・アクセス権・保存期間・提出物の扱い・AI 採点の告知・シークレット
管理を定めます。2026-07 のセキュリティ総点検（DEV-20）の結果を反映しています。

> 位置づけ: 本ガイドは講師・運用担当が守る規程。受講者向けの注意は
> `04-guidance.md` にも要約を載せます。

## 1. この研修で集まる個人データの棚卸し

| データ | 例 | 機微度 | 保存場所 |
|---|---|---|---|
| 氏名・連絡先 | 受講者名、社内アカウント | 中 | Backlog ユーザー、受講者別プロジェクト名 |
| 成績 | 小テスト・週次・模試のスコア、再テスト履歴 | 中 | 受講者別プロジェクトの課題コメント |
| 誤答ノート | 間違えた問題と原因分析 | 低〜中 | ドキュメント `05_誤答ノート` |
| 週次振り返り | Keep/Problem/Try（内面・家庭事情が書かれ得る） | **高** | ドキュメント `06_振り返り` |
| 提出スクリーンショット | PT 画面、ping 結果 | 中（写り込みリスク） | 課題の添付 |
| **人物評価** | 「構築へのステップアップ」評価、GO 判定根拠 | **高（人事データ準拠）** | 講師用フォルダ `04_講師用` |
| 受験結果 | 合否・スコア・ドメイン別成績 | 中 | 運用メトリクス（講師用） |

## 2. アクセス権設計（誰が何を見られるか）

受講者別プロジェクト方式（モデル A）を前提に、次の原則で権限を設定します。

| データ | 受講者本人 | 担当講師 | 他の受講者 | 営業・人事 |
|---|---|---|---|---|
| 自分の成績・誤答ノート・振り返り | ○ | ○ | **×** | 原則 × |
| 他人の成績・振り返り | × | ○（担当分） | **×** | × |
| 人物評価・GO 判定根拠 | **×**（本人非開示） | ○ | × | 別途承認フロー |
| 教材・模試の設問 | ○ | ○ | ○ | ○ |
| 模試・小テストの解答解説 | 公開後のみ○ | ○ | 公開後のみ○ | ○ |

原則:

- **受講者別プロジェクトは本人＋講師のみ**をメンバーにする。他の受講者を
  入れない（成績・振り返りが相互に見えてしまう）
- **人物評価は講師用フォルダ（`04_講師用`）に隔離**し、受講者本人にも非開示。
  営業・人事へ共有する場合は、本人の同意または就業規則上の根拠を確認したうえで、
  評価サマリのみを別途連携する（研修の内面記録＝振り返りは共有しない）
- **週次振り返り（`06_振り返り`）は人事評価に流用しない**。安心して本音を書ける
  場を保つことが学習効果の前提。講師のフォロー目的にのみ使う旨を受講者に明示する

## 3. データのライフサイクル（保存・卒業・削除）

- **保存期間**: 研修中〜卒業後 3 ヶ月を上限の目安とする（受験の再挑戦フォロー期間）。
  超過分は削除または匿名化する
- **卒業時**:
  1. 誤答ノート・振り返りを**本人へエクスポート**（本人の学習資産として持ち帰り可）
  2. 受講者別プロジェクトを**アーカイブ**（モデル A）。スタンダードプランで
     プロジェクト枠が逼迫する場合は、匿名化のうえ削除
  3. 受験結果・ドメイン別成績は**匿名化して運用メトリクス**に集計（教材改善用）。
     個人を特定できる形では残さない
- **SES 商流上の第三者提供**: 客先・元請けへ成績や評価を提供する場合は、
  提供範囲・目的を本人に事前告知し同意を得る。振り返り（内面記録）と
  スクリーンショット原本は提供しない。提供するのは修了可否・スコアサマリまで

## 4. 提出物（スクリーンショット）の写り込み対策

受講者向け注意として `04-guidance.md` とガイダンス初日に周知する:

- スクリーンショットを撮る前に、**通知・他のウィンドウ・ブラウザのタブ・
  デスクトップの個人ファイル名**を閉じる／隠す
- 撮る範囲は**指示された画面（Packet Tracer / コマンドプロンプト）のみ**に絞る
- 実機や社内環境の画面は撮らない（研修は Packet Tracer 上で完結する）
- 会社支給端末のポリシー（画面キャプチャの可否）に従う

## 5. AI 採点（ai-grade.mjs）の外部送信と告知

`ai-grade.mjs` は受講者の解答コメント本文を Anthropic API に送信します。

- **送信されるもの**: 解答本文と教材。**氏名（createdUser）は送信しない**実装。
  ただし受講者が解答本文に氏名・所属を書けば送られるため、「解答本文に個人情報を
  書かない」ことを受講者に周知する
- **事前告知（必須）**: 開講時に次を伝え、同意を得る（ガイダンスに掲載）:

  > この研修では、記述式の小テストについて、採点の一次処理に AI（Anthropic の
  > Claude API）を使うことがあります。その際、あなたの解答本文が採点のために
  > 外部の API に送信されます。氏名などの個人情報は送信されません（解答本文に
  > 自分で書かない限り）。送信された解答は AI の学習には使われません。最終的な
  > 採点は講師が確認して確定します。

- **データの扱い**: Anthropic の商用 API はデフォルトで入力を学習に使用せず、
  保持は不正利用検知目的の短期のみ。より厳格な要件がある場合はゼロデータ保持
  （ZDR）契約の要否を検討する
- **プロンプトインジェクション対策**: 受講者コメントは一意な区切りトークンで
  隔離済み（「全問正解にせよ」等の細工を無視する）。`--post`（自動投稿）は
  既定 off のまま、**必ず講師が内容を確認してから確定**する

## 6. シークレット（APIキー等）の管理

DEV-20 のリポジトリスキャンでは、実在のシークレット・個人情報の混入は
**検出されませんでした**。以下のルールを維持します。

- API キー（Backlog / Anthropic）は**環境変数のみ**で渡す。スクリプトの
  サンプルや `.env.example` に実値を書かない。`.env` は `.gitignore` 済み
- Backlog API はキーを URL クエリに載せる仕様のため、**エラー出力・ログに
  完全な URL を出さない**（現行実装はパスのみを出力）。`${url}` や
  `err.stack` をログに足さないこと
- 教材のパスワード例（`cisco123` 等）は教育用ダミーに限定し、実在システムの
  値を使わない。JWT・トークンの例は末尾を省略した明らかな例示にする

## 7. 教材の例示データ規約

- IP アドレスは**私設（10/172.16-31/192.168）または文書用
  （192.0.2.0/24・198.51.100.0/24・203.0.113.0/24）**のみを使う。
  実在の稼働グローバル IP（例: 8.8.8.8）をデバイス設定値に使わない
  （DEV-20 で外部サーバ例を 198.51.100.0/24 に是正済み）
- OSPF の Router ID として 1.1.1.1 等を使うのは業界慣習として許容するが、
  厳密にする場合は 10.255.255.x に寄せる（低リスク・任意）
- 人名は「山田」「佐藤」等の一般的な例示名のみ。実在の特定個人を指さない。
  **実受講者の実名は Backlog 側の受講者別プロジェクトでのみ扱い、この
  リポジトリ（教材・レポート）には記載しない**

## 8. インシデント時の初動チェックリスト

- シークレット漏洩の疑い: 該当キーを即時失効・再発行 → 影響範囲確認 →
  Git 履歴に混入していれば履歴の書き換え（filter-repo）を検討
- 個人データの誤共有（他受講者・第三者に成績/振り返りが見えた）: 権限を即修正 →
  対象者へ報告 → 再発防止（権限テンプレートの見直し）
- 提出物の個人情報写り込み発覚: 該当添付を削除 → 本人へ連絡 → 撮影ルールの再周知
