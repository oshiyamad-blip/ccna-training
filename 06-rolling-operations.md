# ローリング型運用ガイド（受講者が随時入学・卒業するモデル）

この研修は「1 期生を集めて一斉開講」ではなく、**受講者が随時入学し、20 営業日の
プログラムを終えて卒業していく**継続運用を前提とします。本ドキュメントは
そのための構成と手順を定義します。

## 1. プロジェクト構成 — 2 つのモデル

> **決定（2026-07）: モデル A を採用。** 契約プランは有料（スタンダード = 100
> プロジェクト / プレミアム = プロジェクト無制限）のため、受講者ごとの
> プロジェクト作成が可能。スタンダードプランの場合は、卒業者プロジェクトの
> 定期的な整理（削除または棚卸し）を運用に含めて 100 枠内で回す。

### モデル A: 受講者別プロジェクト方式（推奨・採用）

```
📁 プロジェクト「CCNA-教材」（共有・全受講者が閲覧）
    └─ ドキュメント（ガイダンス / 教材 / ラボ手順書）だけを置く。課題は作らない
📁 プロジェクト「CCNA-山田」 ← 受講者ごとに 1 つ
    └─ 課題 83 件（Week0 P1〜P5 + 本編 Day1〜20 + 試験対策 Day21〜25）・マイルストーン・小テストのやり取り・解答解説
📁 プロジェクト「CCNA-佐藤」
    └─ 同上（開始日が違ってもマイルストーン日付が独立しているので問題なし）
```

**採用理由:**

- **解答のネタバレを構造的に防げる。** 小テストの解答解説・採点コメントは
  受講者本人のプロジェクトにしか存在しないため、後から入学した受講者に
  見えない（単一プロジェクトだと課題・コメントは全員に見える）
- マイルストーン（Week1〜4）の日付が受講者ごとに設定できる
- ガント / ボードがその受講者の進捗だけを表示し、講師が状態を把握しやすい
- 卒業時にプロジェクトごとアーカイブでき、記録が受講者単位で残る

**注意:** プロジェクト数の上限はプランに依存します（上限が小さいプランでは
モデル B を使用）。卒業者のプロジェクトをアーカイブしても上限にカウントされるかは
契約プランの仕様を確認してください。

### モデル B: 単一プロジェクト同居方式（プロジェクトを増やせない場合）

1 つのプロジェクトに全受講者の課題を同居させます。投入時に `--trainee` を指定:

```bash
node scripts/create-backlog-issues.mjs --project CCNA --start 2026-08-17 --trainee 山田 --assignee <userId>
```

- 課題名が `[山田] [Day01] 講義: …`、マイルストーンが `山田-Week1` になり、
  受講者ごとに独立した課題セットが作られる
- 担当者絞り込み・マイルストーン絞り込みで受講者別ビューを作る
- **制約: 課題とコメントは全受講者に見える。** 小テストの解答解説を課題コメントで
  公開すると未受験者に見えてしまうため、次のいずれかで運用する:
  1. 解答解説は受講者に**個別送付**（メール / DM / 個人ドキュメント）
  2. カンニング前提の**性善説運用** + ラボと口頭確認で実力を担保
  3. 小テストを 2〜3 バージョン作りローテーション（教材改訂で対応可能）

## 2. 入学手順（受講者 1 名あたり 30 分）

- [ ] Backlog にユーザー招待（一般ユーザー権限）
- [ ] モデル A: プロジェクト「CCNA-氏名」を作成し、受講者と講師を追加。
      「CCNA-教材」プロジェクトにも受講者を追加（閲覧用）
- [ ] 課題一括投入（開始日 = 入学日の翌営業日以降を推奨。Day00 の準備期間を作る）

  ```bash
  # モデル A
  node scripts/create-backlog-issues.mjs --project CCNA-YAMADA --start 2026-08-17 --assignee <userId>
  # モデル B
  node scripts/create-backlog-issues.mjs --project CCNA --start 2026-08-17 --trainee 山田 --assignee <userId>
  ```

- [ ] 受講者へ案内: 「00_ガイダンス > 研修の進め方」と「Day00 環境構築」の
      リンクを送り、`[Day00] 環境構築` を開始日前日までに終えるよう伝える
- [ ] 誤答ノート用ドキュメント（`05_誤答ノート > 氏名`）の空ページを作成

## 3. 卒業手順

- [ ] 修了テスト（75% 以上）と総合演習ラボの要件充足を確認 →
      **試験対策フェーズ（Day 21〜、`07-exam-phase.md`）へ移行**
- [ ] 模試サイクルで受験 GO 基準を満たしたら受験予約 → **CCNA 合格 = 卒業**
- [ ] 受験結果（合否・スコア・ドメイン別成績）を運用メトリクスに記録
- [ ] 誤答ノートと振り返りを受講者本人へエクスポート（自習で使い続けるため）
- [ ] モデル A: プロジェクトをアーカイブ。モデル B: マイルストーン
      `氏名-Week1〜4` をアーカイブし、課題を「完了」で閉じる
- [ ] 運用メトリクスを記録（下記 6.）

## 4. 週次運用の読み替え（コホート型 → ローリング型）

| コホート型の運用 | ローリング型での読み替え |
|---|---|
| 金曜に全員が週次テスト | **各受講者の Day 5 / 10 / 15 / 20** に個別実施（カレンダー曜日ではない） |
| 月曜朝に全体へ弱点解説 15 分 | 週次テスト翌営業日に**受講者別の個別レビュー 15 分**（誤答 + 振り返りの Try を一緒に確認） |
| 週次で全員の誤答をドメイン集計 | 受講者ごとに集計。全受講者横断の集計は**月次**で行い、教材の弱点（みんなが間違える問題）を改訂に反映 |

## 5. 講師の負荷とスケーリングの目安

受講者はそれぞれ別の Day にいるため、講師業務は「毎日、受講者数ぶんの
採点・確認」になります。

| 同時受講者数 | 講師の目安負荷 | 推奨体制 |
|---|---|---|
| 1〜3 名 | 30〜60 分/日 | 兼務講師 1 名 + 採点スクリプト |
| 4〜8 名 | 1〜2.5 時間/日 | 兼務講師 1 名 + **AI 一次採点の導入を推奨** |
| 9 名〜 | 3 時間超/日 | 専任講師 or AI 採点 Bot + 質問対応の SLA 見直し |

AI 一次採点（記述式の採点案 + 解説コメントの自動生成）を導入する場合は、
Backlog Webhook + Claude API の Bot を追加開発する（`00-feasibility.md` の
拡張案参照）。選択式は `scripts/grade-quiz.mjs` で受講者数によらず一定負荷。

## 6. プログラム自体の改善サイクル

ローリング型は教材が繰り返し使われるため、改善が複利で効きます。

- **記録するメトリクス**（卒業ごと）: 修了テスト点数、週次テストのドメイン別
  正答率、小テストで正答率が 50% を切った設問、質問が集中したトピック、
  本試験の合否（追跡できる場合）
- **月次で教材改訂**: 正答率の低い設問の解説強化・教材該当章の加筆を
  `materials/` に反映 → `upload-wiki.mjs` を再実行すると共有教材が更新される
  （冪等なので何度でも安全）
- 教材の source of truth は常に **この Git リポジトリ**。Backlog 上で直接
  編集せず、リポジトリ → Wiki/ドキュメントの一方向で運用する
